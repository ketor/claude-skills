# team-arch 架构分析技能设计文档

> 日期：2026-03-08

## 概述

创建一个多智能体 Claude Code 技能 `team-arch`，用于对代码仓库进行全面架构分析，通过双分析师独立分析 + 共识合并的方式确保分析准确性，最终输出结构化 Markdown 架构文档和 Mermaid 可视化图表。

## 技能元数据

- **名称**：`team-arch`
- **描述**：启动一个架构分析团队（scanner/analyzer×2/writer），对代码仓库进行全面架构分析，输出结构化架构文档和可视化图表。
- **使用方式**：`/team-arch [--auto] [--depth=shallow|standard|deep] 项目路径或描述`
- **Team 命名**：`team-arch-{YYYYMMDD-HHmmss}`

### 参数

| 参数 | 说明 |
|------|------|
| `--auto` | 自主模式，减少用户确认，仅最终文档需确认 |
| `--depth=shallow\|standard\|deep` | 分析深度，默认 `standard` |

### 分析深度说明

| 深度 | 分析范围 |
|------|---------|
| `shallow` | 仅结构/依赖概览，跳过深入模式分析 |
| `standard` | 结构 + 依赖 + 核心模块架构模式 + 数据流 |
| `deep` | 全量分析包括性能热点、安全边界、演进建议 |

## 角色定义

| 角色 | 职责 |
|------|------|
| **scanner** | 扫描项目结构（目录布局、入口文件、构建系统）、识别技术栈、解析依赖关系（包管理器/import graph）、识别配置文件。**只做结构层面分析，不深入业务逻辑。** |
| **analyzer-1** | 深入阅读核心模块代码，分析架构模式、模块间通信、数据流、错误处理策略。**独立分析，不与 analyzer-2 交流。** |
| **analyzer-2** | 同 analyzer-1 职责，独立执行相同分析。**独立分析，不与 analyzer-1 交流。** |
| **writer** | 对比两位 analyzer 的分析结果，标注一致和分歧之处，合并 scanner 的结构数据，生成 Mermaid 图表，撰写最终架构文档。**不直接阅读代码，只基于分析报告工作。** |

### 关键设计决策

- Analyzer-1 和 analyzer-2 **完全独立分析**，team lead 不让它们互相看到对方的结果，确保共识的客观性
- Writer 负责**对比和合并**，分歧点会在最终文档中标注
- Scanner 和两个 analyzer 可以并行工作

## 流程设计

```
阶段零  项目探测 → scanner 扫描结构 + 技术栈 → 输出项目概况
         ↓
阶段一  并行分析 → scanner 深入依赖分析 + analyzer-1 独立分析 + analyzer-2 独立分析
         ↓
阶段二  共识合并 → writer 对比两份分析 → 标注一致/分歧 → 生成初稿
         ↓
阶段三  分歧仲裁 → team lead 对分歧点让两位 analyzer 各自论证 → 仲裁（或问用户）
         ↓
阶段四  文档生成 → writer 生成最终 Markdown + Mermaid 图表 → 用户确认
         ↓
阶段五  收尾 → 保存文档 + 清理团队
```

### 阶段零：项目探测

- 启动 scanner，快速扫描：目录结构、入口文件、README、构建配置、包管理器
- Scanner 输出项目概况（技术栈、语言、框架、项目规模估算）
- Team lead 据此决定是否调整 `--depth` 建议（超大项目建议 shallow）

### 阶段一：并行分析

- 启动 analyzer-1、analyzer-2，同时 scanner 继续深入依赖分析
- **Scanner**：解析 import/include 图、包依赖、模块间引用关系
- **Analyzer-1 & 2**（独立）：各自阅读核心模块，分析：
  - 架构模式识别（分层/微服务/Monolith/事件驱动/CQRS 等）
  - 模块划分与职责
  - 数据流和状态管理
  - 关键抽象和接口设计
  - `deep` 模式额外分析：性能热点、安全边界、技术债务
- 三者并行，完成后各自向 team lead 发送报告

### 阶段二：共识合并

- 启动 writer，传入 scanner 报告 + 两份 analyzer 报告
- Writer 对比分析：
  - **一致结论**：直接采纳
  - **互补发现**：合并（A1 发现了 A2 没注意的点，反之亦然）
  - **分歧/矛盾**：标注为待仲裁项
- Writer 输出初稿 + 分歧清单

### 阶段三：分歧仲裁

- 如无分歧 → 跳过此阶段
- 有分歧时，team lead 对每个分歧点：
  - 将分歧描述发给两位 analyzer，各自提供论证
  - **标准模式**：重大分歧升级用户决策
  - **自主模式**：team lead 综合论证后裁决，收尾时汇总
- 仲裁结果发给 writer 更新文档

### 阶段四：文档生成

- Writer 生成最终文档，包含：
  - 项目概览（技术栈、规模、构建方式）
  - 架构总览图（Mermaid）
  - 模块关系图（Mermaid）
  - 数据流图（Mermaid）
  - 各模块详解
  - 关键设计决策分析
  - `deep` 模式额外：技术债务评估、演进建议
- Team lead 向用户展示文档摘要，AskUserQuestion 确认

### 阶段五：收尾

- 保存文档到 `docs/architecture/` 目录
- 输出总结（自主模式下汇总所有仲裁决策）
- 关闭所有 teammate，TeamDelete 清理

## 模式对照

| 模式 | 用户确认范围 | 分歧处理 |
|------|-------------|---------|
| **标准模式**（默认） | 项目概况确认 + 分歧仲裁 + 最终文档确认 | 重大分歧升级用户 |
| **自主模式**（`--auto`） | 仅最终文档确认 | team lead 裁决，收尾汇总 |

## 熔断机制

- 如果 analyzer 分歧超过总发现的 50%：暂停问用户（可能分析方向有误）
- 如果项目超大（>10万行）且 depth=deep：建议用户改为 standard

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 项目无法识别技术栈 | Scanner 输出目录结构和文件类型统计，analyzer 基于代码内容推断 |
| 项目过大无法完整分析 | Scanner 识别核心模块，analyzer 聚焦核心模块分析 |
| 两位 analyzer 分析差异极大 | 触发熔断，升级用户确认分析方向 |
| Mermaid 图表过于复杂 | Writer 分层展示（总览图 + 子模块详图），每张图不超过 15 个节点 |

## 最终文档格式

```markdown
# [项目名] 架构分析报告

> 生成时间：YYYY-MM-DD | 分析深度：standard | 共识度：XX%

## 1. 项目概览
- 技术栈、规模、构建方式、入口点

## 2. 架构总览
- 架构模式描述
- Mermaid 总览图

## 3. 模块结构
- 模块列表 + 职责
- Mermaid 模块关系图

## 4. 数据流
- 核心数据流描述
- Mermaid 数据流图

## 5. 依赖分析
- 外部依赖
- 内部模块依赖图

## 6. 关键设计决策
- 识别到的重要设计选择及其 trade-offs

## 7. [deep] 技术债务与演进建议（仅 deep 模式）

## 附录：分析共识说明
- 两位分析师一致结论
- 分歧点及仲裁结果
```

## 技术约束

- 语言无关 — 支持任意编程语言/技术栈
- 技能语言：中文（与 team-dev、team-review 保持一致）
- 遵循现有技能模式：`disable-model-invocation: true`、`$ARGUMENTS` 变量
- 使用 TeamCreate/TeamDelete、TaskCreate/TaskUpdate、SendMessage 等工具
