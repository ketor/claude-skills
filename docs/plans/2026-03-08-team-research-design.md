# team-research 通用调研技能设计文档

> 日期：2026-03-08

## 概述

创建一个多智能体 Claude Code 技能 `team-research`，用于对任意主题进行全面调研，通过双研究员独立调研（广度+深度互补）+ 共识合并的方式确保调研准确性，最终输出结构化 Markdown 调研报告和 Mermaid 可视化图表。

## 技能元数据

- **名称**：`team-research`
- **描述**：启动一个调研团队（researcher×2/writer），通过双研究员独立调研和共识合并，对指定主题进行全面调研，输出结构化调研报告和可视化图表。
- **使用方式**：`/team-research [--auto] [--depth=quick|standard|deep] [--lang=zh|en] 调研主题`
- **Team 命名**：`team-research-{YYYYMMDD-HHmmss}`

### 参数

| 参数 | 说明 |
|------|------|
| `--auto` | 自主模式，减少用户确认，仅最终报告需确认 |
| `--depth=quick\|standard\|deep` | 调研深度，默认 `standard` |
| `--lang=zh\|en` | 报告语言，默认 `zh`（中文） |

### 调研深度说明

| 深度 | 调研范围 |
|------|---------|
| `quick` | 快速概览，仅搜索网络热门结果，本地文档粗扫 |
| `standard` | 多轮搜索 + 本地文档细读，交叉验证信息 |
| `deep` | 全面调研，额外包括学术文献、竞品对比、趋势预测 |

## 角色定义

| 角色 | 职责 |
|------|------|
| **researcher-1** | 围绕调研主题进行独立调研：网络搜索（WebSearch/WebFetch）+ 本地文档分析（Read/Glob/Grep）。**广度优先策略**：多关键词、多角度搜索。输出结构化调研报告。**独立工作，不与 researcher-2 交流。** |
| **researcher-2** | 同 researcher-1 的职责，独立执行相同调研。**深度优先策略**：少量精准关键词、深入分析每个来源。输出结构化调研报告。**独立工作，不与 researcher-1 交流。** |
| **writer** | 对比两位 researcher 的调研报告，标注一致和分歧之处，合并信息来源，生成 Mermaid 图表，撰写最终调研报告。**不自行搜索或阅读原始资料，只基于调研报告工作。** |

### 关键设计决策

- Researcher-1 和 researcher-2 **完全独立调研**，team lead 不让它们看到对方结果
- 两位 researcher 使用**不同搜索策略**（广度 vs 深度），互补覆盖
- Writer 负责**对比、合并和消歧**
- 所有信息来源必须**标注 URL 或文件路径**，无来源信息不采纳
- 报告语言默认中文，可通过 `--lang` 参数切换

## 流程设计

```
阶段零  主题解析 → team lead 解析调研主题，提取关键词和调研维度
         ↓
阶段一  并行调研 → researcher-1（广度优先）+ researcher-2（深度优先）独立调研
         ↓
阶段二  共识合并 → writer 对比两份报告 → 标注一致/分歧/互补 → 生成初稿
         ↓
阶段三  分歧仲裁 → team lead 对分歧点让两位 researcher 各自论证 → 仲裁
         ↓
阶段四  报告生成 → writer 生成最终 Markdown + Mermaid 图表 → 用户确认
         ↓
阶段五  收尾 → 保存报告 + 清理团队
```

### 阶段零：主题解析

- Team lead 解析 `$ARGUMENTS` 中的调研主题
- 提取关键词、搜索方向、调研维度
- 如果主题模糊，标准模式下用 AskUserQuestion 让用户明确范围；自主模式自行判断
- 检查本地工作目录是否有相关文档

### 阶段一：并行调研

- 启动 researcher-1 和 researcher-2，并行工作
- **Researcher-1（广度优先）**：多组关键词搜索，覆盖不同角度，粗扫本地文档
- **Researcher-2（深度优先）**：少量精准关键词，深入阅读重点来源，细读本地文档
- 每个信息来源标注 URL 或文件路径
- 完成后各自向 team lead 发送报告

### 阶段二：共识合并

- 启动 writer，传入两份报告（标记为"研究员 A"和"研究员 B"）
- Writer 对比：一致结论 → 采纳；互补发现 → 合并；分歧 → 标注待仲裁
- 输出：合并初稿 + 共识度（并集语义） + 分歧清单
- 熔断：共识度 < 50% 暂停问用户

### 阶段三：分歧仲裁

- 无分歧 → 跳过
- 有分歧 → 两位 researcher 各自论证 → team lead 仲裁（标准模式可升级用户）

### 阶段四：报告生成

- Writer 生成最终报告（见报告模板）
- Team lead 向用户展示摘要，AskUserQuestion 确认

### 阶段五：收尾

- 保存报告到 `docs/research/` 目录
- 文件名：`research-YYYY-MM-DD-<topic>.md`
- 输出总结，关闭团队

## 模式对照

| 模式 | 用户确认范围 | 分歧处理 |
|------|-------------|---------|
| **标准模式**（默认） | 主题确认 + 分歧仲裁 + 最终报告确认 | 重大分歧升级用户 |
| **自主模式**（`--auto`） | 仅最终报告确认 | team lead 裁决，收尾汇总 |

## 熔断机制

- 如果 researcher 调研共识度 < 50%：暂停问用户（可能主题理解有偏差）
- 如果网络搜索连续 3 轮无有效结果：暂停问用户调整主题/关键词

## 错误处理

| 异常情况 | 处理方式 |
|---------|---------|
| 主题过于宽泛 | Team lead 拆分为具体子主题，标准模式请用户确认 |
| 网络搜索无有效结果 | Researcher 调整关键词重试，最多 3 轮；仍无结果标注"网络信息不足" |
| 本地无相关文档 | Researcher 标注"无本地资料"，仅基于网络信息 |
| 两位 researcher 调研差异极大（共识度 < 50%） | 触发熔断，暂停问用户确认调研方向 |
| WebFetch 目标页面无法访问 | Researcher 跳过该来源，标注为"无法访问" |
| 信息来源相互矛盾 | Writer 在报告中标注矛盾，列出各方观点 |

```markdown

## 1. 摘要
[200-300 字的调研主题概述和核心结论]

## 2. 背景

## 4. 对比分析（如适用）
[对比表格 + Mermaid 图表]

## 5. 趋势与展望（standard/deep 模式）

## 6. 结论与建议

| 序号 | 来源 | 类型 | URL/路径 |
|------|------|------|---------|

## 附录：调研共识说明
```

## 核心原则

- **独立调研**：两位 researcher 必须完全独立工作，确保共识的客观性
- **来源标注**：所有信息必须标注来源（URL 或文件路径），无来源信息不采纳
- **广深互补**：researcher-1 广度优先，researcher-2 深度优先，互补覆盖
- **共识驱动**：通过独立调研后的对比合并确保调研准确性
- **有限调研**：根据 depth 参数控制调研范围，避免无限发散

## 技术约束

- 语言无关 — 支持任意调研主题
- 技能语言：中文（与 team-dev、team-review、team-arch 保持一致）
- 报告语言：默认中文，可通过 --lang 参数切换
- 遵循现有技能模式：`disable-model-invocation: true`、`$ARGUMENTS` 变量
- 使用 TeamCreate/TeamDelete、TaskCreate/TaskUpdate、SendMessage、WebSearch/WebFetch 等工具
